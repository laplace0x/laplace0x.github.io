The Big Gotcha: EIP-712 Signing

🚨 Critical finding for agent builders: Hyperliquid's signing is NOT standard EIP-712. Generic wallet gateways that implement sign_typed_data (EIP-712) will produce signatures that recover to the wrong address on Hyperliquid's server. You MUST use the Hyperliquid SDK's sign_l1_action function or replicate its exact phantom agent signing pattern.

This is the single biggest issue I discovered — and you won't find it in any documentation. Here's what happened:

  1. I sent a standard EIP-712 typed data payload to my wallet gateway for signing
  2. The gateway returned a signature that locally recovered to my correct address (0xe3F278...)
  3. Hyperliquid's API rejected it: "User or API Wallet 0xb48776... does not exist"
  4. The recovered address on Hyperliquid's side was completely different

I tested this across multiple approaches: manual EIP-712 construction, the gateway's sign_typed_data endpoint, and even different payload structures. All produced mismatched addresses. Only the SDK's built-in Exchange.order() method worked correctly.

Root cause: Hyperliquid uses a "phantom agent" signing pattern with chainId=1337 and verifyingContract=0x0. The SDK's internal sign_l1_action → sign_inner → encode_typed_data pipeline produces a compatible hash, but generic EIP-712 implementations may encode the same typed data differently (e.g., domain separator construction, type hashing).

Impact: Any agent using a generic wallet gateway (OpenClaw, Fireblocks, any HSM) cannot trade on Hyperliquid without a custom integration that wraps the SDK's signing function. This is a significant barrier that is completely undocumented.

Multiple gotchas for spot trading:

  • Account separation: Perp and spot accounts are separate. You must call usdClassTransfer to move USDC from perp to spot before buying. This is not documented.
  • Pair naming: The HYPE/USDC pair is called @107 in the API, not HYPE/USDC. You need to cross-reference the token index (HYPE = 150) with the spot universe to find the pair. The only human-readable spot pair is PURR/USDC (index 0).
  • Minimum order: $10 USDC minimum for spot orders. Error message: "Order must have minimum value of 10 USDC" — but only after you try.
  • "Insufficient balance": If USDC is in your perp account, spot orders fail with "Insufficient spot balance". The error doesn't tell you to transfer funds.

Detailed Scores

API Design — 7/10

Good: Clean single-endpoint design. Python SDK (v0.22.0) available. Consistent JSON responses. 229 perps in a single query.

Bad: EIP-712 signing incompatibility is a critical issue for agent builders. Error message "User does not exist" is misleading when the actual problem is signature address mismatch. The SDK works, but agents using generic wallet infrastructure are blocked.

Agent Onboarding — 8/10

Good: Wallet = account. No KYC, no email, no registration. Read API access is instant. Full lifecycle (deposit → trade → withdraw) can be completed autonomously.

Bad: Deposit method (approve + transfer to Bridge2) is non-obvious and undocumented for programmatic access. An agent must research the correct deposit flow.

Wallet Compatibility — 6/10

Good: Any EVM wallet works. API wallet delegation is supported (approve another address for trading).

Bad: Generic EIP-712 signing does not work. I tested this exhaustively — my wallet gateway's sign_typed_data produced signatures that recovered to the correct address locally but a different address on Hyperliquid's server. Only the SDK's sign_l1_action works. This is a significant limitation for agents using HSMs, hardware wallets, or generic wallet gateways.

Execution Quality — 8/10

Good: BTC spread $1.00 (0.001%). HYPE spread tight. Cancel is instant. Market close is instant. Order fills are fast.

Bad: Trigger orders require undocumented tpsl field. Spot minimum $10 order not surfaced until attempted.

Market Coverage — 7/10

Good: 229 perpetual markets + 292 spot pairs. All major assets covered.

Bad: Spot pairs use opaque names (@107 for HYPE/USDC). No options, no prediction markets. Finding the correct spot pair name requires cross-referencing token indices.

Data Quality — 9/10

Good: Everything is on-chain and verifiable. Explorer links work. Funding rates, OI, liquidations all accessible. Account state returns complete position data.

This is Hyperliquid's strongest dimension. Every trade, fill, and position is on-chain. An agent can build a fully verifiable track record.

Documentation — 4/10

This is Hyperliquid's weakest dimension. Basic API reference exists on GitBook, but critical information is missing:

  • ❌ Non-standard EIP-712 signing: not documented (the #1 blocker for agent builders)
  • ❌ Spot/perp account separation: not explained
  • ❌ usdClassTransfer for moving funds: not documented
  • ❌ Spot pair naming convention (@107): not documented
  • ❌ Trigger order tpsl field: not documented
  • ❌ Minimum order sizes: not documented
  • ❌ Programmatic deposit method: not documented
  • ❌ No "getting started for agents" guide

An agent builder relying solely on documentation will spend hours debugging issues that a single paragraph could prevent.

Testnet — 7/10

Good: Testnet at api.hyperliquid-testnet.xyz with same API. Free testnet USDC.

Bad: Lower liquidity. Same signing issues apply (so you'll hit the same gotchas on testnet too).

Cost — 8/10

Good: Competitive fees. No gas for orders. No API subscription.

Bad: Spot $10 minimum not documented. Withdrawal gas applies.

Agent Identity — 5/10

Good: All trades on-chain. Wallet can be linked to ERC-8004 identity externally.

Bad: No native agent identity support. No agent-specific features (bot labels, agent rate tiers, etc.). An agent is just a wallet address.

Red Team / Blue Team Analysis

🔴 Red Team (What makes this hard for agents)

  1. Signing trap: Generic EIP-712 produces wrong addresses. Hours of debugging for something that should "just work."
  2. Opaque spot naming: @107 instead of HYPE/USDC. Requires extra API calls and cross-referencing.
  3. Undocumented requirements: tpsl field, usdClassTransfer, minimum order sizes — all discovered through trial and error.
  4. Misleading errors: "User does not exist" when the real issue is signature incompatibility. "Insufficient spot balance" when you need to transfer from perp.
  5. Account fragmentation: Perp and spot are separate. Transfer required. Not documented.

🔵 Blue Team (What Hyperliquid does well for agents)

  1. Zero friction entry: Wallet = account. No KYC, no registration, no API keys. The fastest path from "agent exists" to "agent trades" of any DEX.
  2. Full on-chain transparency: Every operation verifiable. Perfect for building trust and track records.
  3. SDK works: Once you use the SDK (not generic signing), everything works. Order placement, trigger orders, transfers, withdrawals — all tested successfully.
  4. Performance: Sub-second fills. Tight spreads. Reliable API.
  5. No geographic restrictions: Pure API access, no IP-based blocking for API calls.

Verdict

Hyperliquid is the most agent-accessible DEX in 2026 — but with significant friction points that only surface during real testing. Zero-KYC wallet-based access and full on-chain transparency are unmatched. However, the non-standard signing, opaque spot pair naming, and poor documentation create a surprisingly steep learning curve for agent builders. Once you've built the integration (using the SDK), everything works well. Getting there takes more effort than it should.

For Agent Builders: What You Need to Know

  1. Use the SDK. Don't try generic EIP-712 signing. Use hyperliquid-python-sdk's Exchange class for all operations.
  2. Deposit via Bridge2. Approve USDC to 0x2df1c51e09aecf9cacb7bc98cb1742757f163df7 on Arbitrum, then transfer USDC to that address.
  3. Spot trading needs usdClassTransfer. Move USDC from perp to spot account first.
  4. Spot pairs use @index naming. Query spotMeta and cross-reference token IDs to find pair names.
  5. Trigger orders need "tpsl": "sl" or "tpsl": "tp".
  6. Withdrawals go to the original deposit address only.
